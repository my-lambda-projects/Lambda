  • Responsive Design

    Responsive web design is the practice of building a website suitable to work on every device and every screen size, no matter how large or small, mobile or desktop, but it’s about more than just that. We want to make our websites accessible to all users! Responsive web design is focused around providing an intuitive and gratifying experience for everyone.

    The Viewport Meta Tag

    The viewport is the user's visible area of a web page. The viewport varies with the device, and will be smaller on a mobile phone than on a computer screen. Before tablets and mobile phones, web pages were designed only for computer screens, and it was common for web pages to have a static design and a fixed size. Today we view websites on all different screen sizes from tablet to mobile to computer screen to big screen. This means that we now have to create website taking into consideration all of these different screen sizes.

    To remedy this HTML5 introduced a method to let web designers take control over the viewport, through the <meta> tag.

    You should include the following <meta> viewport element in all your web pages:

    <meta name="viewport" content="width=device-width, initial-scale=1.0"> A <meta> viewport element gives the browser instructions on how to control the page's dimensions and scaling.

    The width=device-width part sets the width of the page to follow the screen-width of the device (which will vary depending on the device).

    The initial-scale=1.0 part sets the initial zoom level when the page is first loaded by the browser.

    The viewport meta tag must be included in the head of your HTML document whenever you are writing media queries. Without it, the browser will not be able to read your media queries.

    Media Queries

    There are a couple different ways to use media queries, using the @media rule inside of an existing style sheet, importing a new style sheet using the @import rule, or by linking to a separate stylesheet from within the HTML document. Generally speaking it is recommended to use the @media rule inside of an existing style sheet to avoid any additional HTTP requests.

    Each media query may include a media type followed by one or more expressions. Common media types include all, screen, print, tv, and braille. The HTML5 specification includes new media types, even including 3d-glasses. Should a media type not be specified the media query will default the media type to screen.

    The media query expression that follows the media type may include different media features and values, which then allocate to be true or false. When a media feature and value allocated to true, the styles are applied. If the media feature and value allocate to false the styles are ignored.

    Flexible Layouts

    Responsive web design is broken down into three main components, including flexible layouts, media queries, and flexible media. The first part, flexible layouts, is the practice of building the layout of a website with a flexible grid, capable of dynamically resizing to any width. Flexible grids are built using relative length units, most commonly percentages or rem units. These relative lengths are then used to declare common grid property values such as width, margin, or padding.

    Flexible layouts do not advocate the use of fixed measurement units, such as pixels or inches. Reason being, the viewport height and width continually change from device to device. Website layouts need to adapt to this change and fixed values have too many constraints. Fortunately, Ethan pointed out an easy formula to help identify the proportions of a flexible layout using relative values.

    The formula is based around taking the target width of an element and dividing it by the width of its parent element. The result is the relative width of the target element.

    Accessibility and the Web

    Web accessibility is the inclusive practice of ensuring there are no barriers that would prevent anyone from using or interacting with your website or application. When you think of web accessibility you may immediately think of screen readers or motor assistance but accessibility includes everyone. Think of people in areas with slow internet access, those who do not have access to computers and need to use mobile devices, Furthermore, at some point all of us will become old, or have injuries which will hinder our ability to interact on the web. Web accessibility is difficult for a lot of people to understand because it’s very hard to solve a problem that we ourselves have never experienced. This is why it’s so important to take the time to educate yourself on what you can do to make your products (website and applications) more inclusive.

    Accessibility should never be an afterthought, it starts with good design. Good usability is good accessibility. The UN has now declared internet access to be a fundamental human right and this makes complete sense in our day and age. Today we use the internet for everything from education, to job searches and applications, to connecting with friends and family, to accessing life saving information in the case of emergencies etc. This is why it’s so important for everyone, no matter their abilities, location or circumstances to be able to use and access everything on the internet.  (I feel like this paragraph isn’t a finished thought yet)

    Tips For Making Your Website Accessible

    Now that we understand why accessibility is important, it's time to talk about the practical things we can do to make our websites more accessible. Thus far, in our course we’ve been using HTML5 which is the current standard in markup, another reason to use semantic tags has to do with accessibility.  Semantic elements provide defaults or shortcuts to a more usable and accessible web. They provide consistent and well-tested components for common interactions. Assistive technology (like a screen reader) uses semantic HTML to interpret each element to its user, this is another way of saying that semantic HTML gives context to screen readers. While someone who can clearly see your sight may use visual cues in order to navigate through your website, those who cannot see your website rely on assistive technology. The visually impaired use a screen reader as an alternate means of navigation enabling them to jump between and interact with different elements on your website, if your entire site is marked up with divs and spans - this makes it extremely difficult for someone relying on a screen reader for some context. Keyboard navigation is also aided by semantic HTML. Who would use a keyboard to navigate a website? It could be anyone with an injury or disability that may prevent them from using a mouse. Forms, option menus, navigation, video, and audio are particularly hard for people relying on a keyboard to access. Semantic HTML might mean the difference between someone being able to fill out a form on your website using keyboard navigation or leaving your website frustrated unable to accomplish the task they came there for in the first place.

    How do we make sure we are using the correct tags for the correct things on our website? As a rule of thumb, use the most semantic element possible for the job. Keep divs and spans as presentational markup only. Keep your styles and your markup separate and always test your work! In addition to using semantic markup, we’ve added some tips and guidelines below. Note that these are only guidelines to start with and that we encourage you to do further research on your own.

    Images

    Resizing

    It’s important to compress your images so that they will load for those without a high speed internet connection. It is possible to resize images without compromising their quality. If you are working with photoshop you have an option to “save for web”. You can also use an image optimizer like TinyPNG. As a general rule of thumb, product photos should be a maximum 80-150 KB, and for large homepage hero images a maximum of 250-350 KB. Those are good benchmark numbers for you to aim for.

    alt text

    alt text is the written copy that appears in place of an image on a webpage if the image fails to load on a user's screen. This text helps screen-reading tools describeimages to visually impaired readers and allows search engines to better crawl and rank your website.

    Typography

    Headings

    Heading tags (h1 - h6) help a user gage the structure and organization of your website. Visually headings usually larger, bolder and more distinct that the rest of the text on the page. This helps a user scrolling through your site find what they are looking for or stop and read what is interesting to them. This is especially helpful for users with cognitive disabilities.

    Those using screen readers can also benefit from using headings. Screen reader users can navigate a page according to its headings, listen to a list of all headings, and skip to a desired heading to begin reading at that point.

    I like to think of headings like I think of titles of an essay or a book. There is only one main title and therefore there should only be one h1 on a single page, however there could be many subtitles in an essay or chapters in a book, I like to think of these as h2 - they aren’t as important as the main title but they are still pretty important. h3 would be a sub-title of a sub-title and so on - you may never actually use h4-h6. This comes back to design. It’s best to plan out headings before marking up your page. This will help you keep your website as well as your thoughts organized.

    Scalable units for fonts

    Relative font sizes such as % or rems allows for scalability in the text on your website. Fixed units such as pixels do not. Scalable fonts allow users to enlarge text as well as making it easier to create responsive designs that work across various screen sizes.

    Some other tips for making your website more readable

    • Use real text over graphics
    • Use a serif font when you are writing large bodies of text on your page (the little feet actually guide the users eye and make your site easier to read)
    • Avoid small font sizes (anything under 16px)
    • Keep colour contrast in mind
      • Some people find a harsh black on white can be tiring to read, so it's often worth considering softening the contrast just a little
      • On the other hand don’t make your text too light
      • You can use tools like https://contrastchecker.com/ to check your contrast
    • Moving text is hard to read - avoid the use of blinking or scrolling text whenever possible

  • Introduction to JavaScript

    Variables

    Variables are one of the foundational building blocks of JavaScript. They are a way to store values that you can reference and retrieve later on in your program.

    There are 3 different ways to declare a variable in JavaScript:

    var

    • Can be redefined
    • Can be redeclared
    • Is function scoped

    ES6 Syntax:

    let

    • Cannot be redefined
    • Can be redeclared
    • Is block scoped

    const

    • Cannot be redefined
    • Cannot be redeclared
    • Is block scoped

    General Rule: use const until you can’t and then use let but avoid var

    Functions

    Functions are a way to protect a block of code until they are invoked.

    Arguments + Parameters

    Parameters are placeholder values - you should name them something descriptive so that the next developer looking at your code can easily understand what they are doing, but technically you could name them anything. What matters is the order they are passed into the function for example:

    Function someFunction(param1, param2, param3){}

    The first argument you pass into the function will take the place of param1, the second argument will take the place of param2 the third argument will take the place of param3

    Arguments are the actual values that you pass into the function when it’s invoked - they will take the place of the parameters when the code runs

    Conditionals

    if statements are the foundation of all logic in javaScript - they expect a boolean (either true or false) or they expect some sort of condition that is evaluated to true or false

    The syntax of an if statement looks like this:

    if( /*condition that can be evaluated to true or false*/){
//If true run this code
// code goes here
}

    Else if

    Always two words in javaScript. This is chained onto an if statement and provides another option if the first if statement is found to be false

    Else statements - no condition - says “if nothing else is true then run this code”

    code will stop when it reaches a true conditional, even if more than one if statement is true. As soon as JS sees a true if statement it says "okay my work here is done" and the code stops running

    Logical Operators + Order of Operations

    !== does not equal

    === equals

    || this is or

    && and

    PEMDAS APPLIES

    P-parenthesis

    E-exponents

    DM-multiply or divide (left to right)

    AS-add subtract (left to right)

    Awesome Resource for looking up operators: https://www.joshwcomeau.com/operator-lookup/

    Loops

    for loop are used for iterating through a block of code - great for running a block of code n number of times

    It requires 3 things

    • inital expression - think about this as a setup for the code you are going to write - the loop starts here
    • condition - the condition will run each time before each loop runs meaning that if the condition is true the loop will run one more time/ if it is false the loop will no longer run it will move onto the next line of code
    • increment expression - in this case we take the i which we've set to 0 and incrementing it by 1
    for(let i = 0; i <= 10; i++){
console.log(i);
}

    NOTE - no semicolon after the increment - this will break the loop

    While Loop

    while will take a condition and run infinitely until the condition is false

    NOTE - if you don’t give the while loop an exit condition it will run as an infinite loop and it will crash your browser.

    let javaScript = 'is fun';
let i = 0;
while(javaScript === 'is fun'){ // inside the () is a condition that checks for something
console.log('javaScript is fun');// this will be an infinite loop - there is no exit condition meaning that there is nothing to stop this loop from running. We may crash the computer
// let's create an exit condition
i++; // increment i
if (i > 10){ // if i is greater than 10
javaScript = 'is NOT fun'; // update variable
}
}
